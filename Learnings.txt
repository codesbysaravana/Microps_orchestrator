### How this architecture works seamlessly:
1. **The Build:** Jenkins finishes the build and pushes `poopdi:build-123` to ECR.
2. **The Blueprint:** `EcsDeployService.js` creates a new "Task Definition" (a JSON blueprint) in AWS saying: *"Hey AWS, the latest version of User 11's app is located at ECR URI `...poopdi:build-123`"*.
3. **The Rollout:** The script tells the user's existing AWS ECS Service to update. AWS Fargate will automatically provision a micro-VM, pull the image from ECR, start the Next.js app on port 3000, and once it's healthy, shut down the old version.

### The Missing AWS Prerequisite (For your next tests):
For this specific Node script to succeed, you have to create a few things in the AWS Console *once* before your first user deploys:
1. **ECS Cluster:** Create a blank AWS ECS Cluster named `microps-tenant-cluster` (Fargate mode).
2. **Execution Role:** An IAM Role that allows ECS to pull images from your ECR vault and write logs to CloudWatch.
3. **The First Service:** For your upcoming tests, you'll need to manually create the first Service (`tenant-11-poopdi-service`) in that cluster so the script has something to update. Later, you can have Node.js use `CreateServiceCommand` to automate this for brand-new users.

How to get those 3 IDs fast for your .env:
VPC Dashboard: Open the AWS Console and search for VPC.
Subnets: Click Subnets on the left. You will see default subnets listed. Copy any two of the Subnet IDs (e.g., subnet-0a1b2c3d4e5f6g7h8).
Add them to your .env as ECS_SUBNET_1=... and ECS_SUBNET_2=....
Security Group: Click Security Groups on the left. Look for the one named default (or create a new one that allows port 3000 / HTTP inbound). Copy its ID (e.g., sg-0123456789abcdef0).
Add it to your .env as ECS_SECURITY_GROUP=....

DONT READ ALL THIS:

Right now, your ECS service spins up, but it is floating in the void. To route traffic from the internet -> ALB -> Target Group -> Fargate Task, your Node.js orchestrator needs to dynamically create Target Groups and ALB Listener Rules on the fly before it creates the ECS service.

Here is the fully upgraded deployment engine. I have integrated the AWS ELBv2 SDK. For every new user, it will now automatically create a Target Group, map a unique subdomain (like tenant11-poopdi.microps.com) to an ALB Listener Rule, and attach it to the Fargate container.

1. You will need to install the load balancer SDK first: npm install @aws-sdk/client-elastic-load-balancing-v2
2. How to set up your AWS environment for this:

The ALB: Manually create an Application Load Balancer in the AWS EC2 dashboard. Attach it to your public subnets and ensure its securiy group       allows inbound HTTP (Port 80) and HTTPS (Port 443).

The Listener: On your ALB, add a Listener on Port 80 (and 443 later). Set the default action to return a fixed response (e.g., 404 Not Found - MicrOps).

The Envs: Grab the VPC ID and the Listener ARN (from the listener you just made on the ALB) and put them in your .env alongside your subnets.

Wildcard DNS (Crucial): Go to your domain registrar (e.g., Route53, Namecheap, Vercel) and create a Wildcard CNAME Record (*.microps.com) pointing to your ALB's DNS Name.
