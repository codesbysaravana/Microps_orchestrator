📜 PART 1: What was going wrong previously (The Roadblocks)
During your initial testing, four distinct bottlenecks blocked your deployment:

1. Local Laptop Crash (EADDRINUSE :::5000)
What happened: When you ran node server in your MINGW64 terminal, Node threw an unhandled error and crashed.
Why: Port 5000 on your Windows machine was already occupied by another background Node process.

2. CloudWatch Startup Crash (AccessDeniedException)
What happened: Your first ECS Fargate container failed immediately on boot with ResourceInitializationError.
Why: Your Task Definition instructed AWS to auto-create log group /ecs/microps-backend ("awslogs-create-group": "true"). However, your IAM role (microps-ecs-execution-role) lacked the logs:CreateLogGroup IAM policy permission.
The Fix Applied: We manually provisioned /ecs/microps-backend via the CLI so ECS didn't need creation permissions.

3. Orchestrator Firewall Blackhole (connectex: Connection Timed Out)
What happened: Once Fargate task #1 reached RUNNING, browser requests to http://13.210.218.184:5000/health hung infinitely and timed out.
Why: AWS Security Group sg-05f89ba... defaulted to blocking inbound public HTTP requests on TCP port 5000.
The Fix Applied: We added Inbound Rule sgr-004bd70a... allowing TCP port 5000 from 0.0.0.0/0.

4. Tenant ALB Health Check Trap (sgr-0a128d5...)
What happened: In DeployService.js, dynamic tenant containers on port 3000 were configured to accept traffic only from other resources sharing Security Group sg-05f89ba....
Why: Because your Application Load Balancer (ALB) operates inside its own separate security group, ALB target group health checks were getting dropped at the firewall.
The Fix Applied: We authorized TCP port 3000 inbound so ALB probes return Healthy (200 OK).
